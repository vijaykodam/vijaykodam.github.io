# 3D Wrap Preview

Page: `web/3d/index.html` + `web/3d/app.js`. Loads the active model's
`web/models/<id>/model.glb`, replaces the diffuse map of any material
named like `body`, `paint`, or `wrap` with the user's wrap PNG (passed
via `sessionStorage` key `tesla_wrap_dataurl`), and renders with
OrbitControls.

## Procedural pipeline (preferred)

Each model.glb in this repo is the **output of a checked-in Python
script**, not a hand-edited binary asset. Scripts live in
`scripts/blender/build_<model>.py` and are run inside Blender (or
through the Blender MCP server) to regenerate the GLB.

To rebuild the Cybertruck GLB:

```bash
# Option A: via the Blender Python API directly.
/Applications/Blender.app/Contents/MacOS/Blender \
  --background --python scripts/blender/build_cybertruck.py

# Option B: via the Blender MCP server with Blender already open.
# Send the file's contents to mcp__blender__execute_blender_code.
```

The script is idempotent. Output: `web/models/cybertruck/model.glb`.

After regenerating, flip `web/models/manifest.json` ->
`<id>.hasGlb` to `true` if it isn't already, and the viewer will
pick it up.

### Why scripts, not Sketchfab GLBs

- No third-party license surface — every vertex is generated by our code.
- Reproducible: anyone with Blender + this repo can rebuild any GLB.
- Diffable: changes to the mesh show up as Python diffs in PRs.
- Lets us guarantee the body material name (`body_paint`, matches
  `BODY_MATERIAL_HINTS` in `app.js`) and the UV layout that the
  template.png expects, instead of fighting an artist's UVs.

### Caveats / status

- v1 of `build_cybertruck.py` ships an extruded side-profile body with
  smart-projected UVs (not yet template-aligned). The wrap appears on
  the body, but panel boundaries don't line up with the cybertruck's
  `template.png` regions yet. Iterating UV island placement to overlay
  the template panels is tracked as Phase 5.2 deferred work in TODO.md.
- v1 has no separate window mesh — windows render as body paint. Adding
  a `windows` material is also tracked in TODO.md.

## Manual fallback (for sourced meshes)

Use this when you'd rather adapt an existing third-party Tesla mesh
than write/extend a procedural builder.

## Why per-model GLBs need manual UV prep

Tesla's `template.png` files are unfolded panel layouts where pixels inside
panels are gray-shaded and pixels outside are white. A wrap PNG only maps
correctly onto a 3D Tesla model if that model's body-paint mesh has UVs
that match this exact unfolded layout. Off-the-shelf Sketchfab Tesla
meshes will not have this — they ship with whatever UVs the original
artist used (or none).

So adding a new model is a Blender task, not a code task.

## Pipeline (per Tesla model)

1. **Source the mesh.** Find a free / appropriately-licensed Tesla GLB
   for the model id you're targeting. Cybertruck is the easy first win
   because Tesla itself ships a Cybertruck mesh in some asset packs;
   for Model Y, Sketchfab "Tesla Model Y" CC-BY meshes are common.

2. **Open in Blender.** Import the GLB. Identify the body-paint mesh
   (usually called `body`, `chassis`, `outer_shell`, etc.) and the
   material assigned to it.

3. **Rename the body material to include `body`, `paint`, or `wrap`** so
   `app.js` recognizes it via the `BODY_MATERIAL_HINTS` substring match.
   E.g. `body_paint` is fine.

4. **Re-UV the body-paint surfaces** so they unfold to match the
   layout in `web/models/<id>/template.png`:
   - In UV editing mode, load `template.png` as a background reference
     in the UV editor (UV Editor > N panel > Image).
   - Mark seams that approximate the panel boundaries Tesla uses (hood
     seam, door seams, roof, etc.).
   - Unwrap (`U > Unwrap`).
   - Manually scale/rotate/translate UV islands so each panel island
     overlaps the corresponding shape in `template.png`.
   - This is the most time-consuming step. Plan ~1-3 hours per model.

5. **Test inside Blender.** Apply `template.png` as the diffuse texture
   on the body material. The unfolded panels should appear in the right
   places when rotating the model. Ship-quality is when a known wrap
   (e.g. Racing_Stripes.png) reads correctly across hood, doors, roof.

6. **Export.** `File > Export > glTF 2.0 (.glb)`. Settings:
   - Format: `glTF Binary (.glb)`
   - Include: Selected Objects (the body + minimum visual extras)
   - Materials: Export, Images: Automatic
   - Compression: Off (we serve raw, the in-car browser handles ~5 MB
     without trouble; Draco is overkill for one model)

7. **Drop the result** at `web/models/<id>/model.glb`.

8. **Flip the manifest flag.** In `web/models/manifest.json`, set
   `"hasGlb": true` for that model id. Optionally set `"glbUrl"` to a
   filename other than the default `model.glb`. This makes the
   `View in 3D` CTA appear in the kid topbar and on the landing page.

9. **Test in browser.** Navigate to `kid/?model=<id>`, paint a wrap,
   click `View in 3D`. The wrap should appear on the body. If the
   material name didn't match a hint, open the browser console — `app.js`
   logs a warning naming the materials it saw.

## Adjusting the body-material hints

`web/3d/app.js` uses `BODY_MATERIAL_HINTS = ['body', 'paint', 'wrap']`.
If a particular GLB uses an unusual material name (e.g. `M_Exterior`),
either rename the material in Blender (preferred — keeps the matcher
simple) or extend the array.

## Performance notes

- `setPixelRatio(min(2, devicePixelRatio))` caps retina cost on phones.
- `THREE.CanvasTexture` of a 1024x1024 PNG with anisotropy applied
  tested at 60 fps on M1 Air laptop, ~30 fps on a 2021 iPhone.
- Tone mapping is `ACESFilmicToneMapping`, exposure 1.0. Increase
  exposure if a particular GLB ships with very dark materials.
- An RGBELoader environment map would improve chrome reflections;
  intentionally omitted for now to keep CDN fetch surface small.

## Offline build (vendored)

Three.js is vendored under `web/3d/vendor/three/` so the page works
without a CDN — required for the flaky Tesla in-car LTE. The importmap
in `index.html` resolves `three` and `three/addons/` to the local copies.

What's vendored (1.4 MB total, only the closure `app.js` actually imports):

```
web/3d/vendor/three/
  three.module.js
  addons/controls/OrbitControls.js
  addons/loaders/GLTFLoader.js
  addons/utils/BufferGeometryUtils.js   # transitive dep of GLTFLoader
```

To bump the version (currently `three@0.160.0`):

```bash
mkdir -p /tmp/three-pack && cd /tmp/three-pack
npm pack three@<version> && tar -xzf three-<version>.tgz
cd /Users/vg/experiments/tesla
cp /tmp/three-pack/package/build/three.module.js web/3d/vendor/three/
cp /tmp/three-pack/package/examples/jsm/controls/OrbitControls.js \
   web/3d/vendor/three/addons/controls/
cp /tmp/three-pack/package/examples/jsm/loaders/GLTFLoader.js \
   web/3d/vendor/three/addons/loaders/
cp /tmp/three-pack/package/examples/jsm/utils/BufferGeometryUtils.js \
   web/3d/vendor/three/addons/utils/
```

If a new addon is needed (e.g. `RGBELoader` for env maps), copy it under
the matching `addons/<dir>/` and check its imports — pull in any new
transitive deps the same way.
